# NTU Gym Tracker — 實作計畫書

> 學校健身房即時人數的「收集 → 展示 → 預測」平台。
> 定位：**給全校同學使用的正式服務**（需考慮穩定部署、長期維運）。
> 撰寫日期：2026-06-29

---

## 1. 專案目標與範圍

打造一個服務，能：

1. **定時收集**學校健身房的即時人數，建立長期歷史資料。
2. **公開網站**讓所有同學查詢歷史趨勢（日/週/月平均、時段熱力圖）。
3. **預測**未來人數，幫助同學避開人潮、挑空檔運動。

### 成功標準（MVP）
- 連續穩定收集資料 ≥ 4 週、缺漏率 < 5%。
- 網站可顯示「本週各時段熱力圖」與「今日即時人數」。
- 預測 baseline 的 MAE 優於「全體平均」這個 naive 基準。

### 非目標（先不做）
- 帳號系統 / 個人化。
- App（先做 RWD 網頁即可）。
- 多場館（先鎖定單一健身中心，架構保留擴充欄位 `venue_id`）。

## 2. 系統架構總覽

```
┌─────────────┐   每 N 分鐘    ┌──────────────┐
│  Scheduler  │ ────────────▶ │   Scraper    │
│ (cron/      │               │ (抓取+解析)  │
│  GitHub     │               └──────┬───────┘
│  Actions)   │                      │ 寫入
└─────────────┘                      ▼
                              ┌──────────────┐
                              │   Database   │
                              │ (SQLite →    │
                              │  Postgres)   │
                              └──────┬───────┘
                          讀取       │        讀取
                   ┌─────────────────┼─────────────────┐
                   ▼                                   ▼
            ┌──────────────┐                    ┌──────────────┐
            │  Web / API   │                    │  ML Pipeline │
            │ (FastAPI +   │                    │ (訓練+預測)  │
            │  前端圖表)   │                    └──────────────┘
            └──────────────┘
```

**技術選型（Python 3.12，沿用現有 `uv` / `pyproject.toml`）**

| 元件 | 選擇 | 理由 |
|------|------|------|
| 套件管理 | `uv` | 專案已採用 |
| 爬蟲 | `httpx` + `selectolax`/`beautifulsoup4` | 輕量、快；若是 JS 渲染才考慮 `playwright` |
| 排程 | 先 GitHub Actions cron（免費、免顧機器），上線後可換成主機 cron / APScheduler | 降低維運成本 |
| 資料庫 | 先 **SQLite**，資料量/併發變大再遷移 **Postgres** | 起步零成本，schema 不變好遷移 |
| 後端 API | **FastAPI** + `uvicorn` | 非同步、自動產生 OpenAPI 文件 |
| 前端 | 先 Jinja2 + 圖表庫（Chart.js / ECharts）或 HTMX；視覺化原型可用 Streamlit | 求快、求穩 |
| ML | `pandas` + `scikit-learn`（baseline）；進階再 `lightgbm` | 週期性資料先用樹模型，不急著上深度學習 |
| 部署 | 容器化（Docker）+ 雲端小主機 / Fly.io / Railway | 正式服務需 24/7 |

## 3. 模組一：資料收集（Scraper）

### 3.1 先探查資料來源（你目前在這一步）
因為「知道網址但還沒研究結構」，第一個任務是**逆向理解資料怎麼來的**：

1. 打開瀏覽器 DevTools → **Network 分頁**，重新整理頁面。
2. 找出人數數字是怎麼來的：
   - **情況 A：背後有 JSON/XHR API** → 最理想。直接打那個 endpoint，最穩定、最省資源。記下 URL、參數、回傳格式。
   - **情況 B：人數寫死在 HTML** → 用 `httpx` 抓頁面 + CSS selector 解析。
   - **情況 C：iframe / 第三方即時看板（如 人流統計廠商）** → 追到 iframe 的真實來源網址，多半又回到 A 或 B。
   - **情況 D：純 JS 動態渲染、抓不到資料** → 才用 `playwright` 無頭瀏覽器（成本最高，最後手段）。
3. 確認**更新頻率**：網站本身多久更新一次？我們的抓取頻率不該高於它（否則只是重複資料）。

### 3.2 抓取策略與禮貌原則（正式服務必做）
- 遵守 `robots.txt`；設定可識別的 `User-Agent`（含聯絡方式）。
- 頻率建議 **每 5–10 分鐘一次**（多數人不會分鐘級別在意，且足夠做小時預測）。
- 加上 timeout、重試（指數退避）、失敗告警。
- **網站當機/抓取失敗時，資料留「空（null）」而非補 0**，避免污染分析。

### 3.3 資料模型（schema）
一筆觀測 = 一列（long format）：

| 欄位 | 型別 | 說明 |
|------|------|------|
| `id` | int | 主鍵 |
| `venue_id` | str | 場館代號（預留多場館） |
| `scraped_at` | datetime (UTC) | **實際抓取時間**（存 UTC，顯示再轉 Asia/Taipei） |
| `count` | int / null | 當下人數；失敗為 null |
| `capacity` | int / null | 場館上限（若網站有提供） |
| `source_status` | str | `ok` / `timeout` / `parse_error` 等 |

> 衍生特徵（星期、時段、是否假日…）**不存進原始表**，留到分析/訓練時再算，保持原始資料乾淨。

### 3.4 儲存格式討論
- **不建議**一堆零散 JSON 檔（難查詢、難聚合）。
- 建議 **SQLite**（單檔、好備份、SQL 直接聚合）；若要做大量離線分析可額外匯出 **Parquet**。
- 每日自動備份資料庫檔（正式服務的資料是最有價值的資產）。

## 4. 模組二：網站與 API

### 4.1 API（FastAPI）
- `GET /api/current` — 最新一筆人數。
- `GET /api/history?from=&to=&granularity=hour|day` — 區間歷史（已聚合）。
- `GET /api/heatmap` — 星期 × 時段 的平均人數矩陣。
- `GET /api/forecast?date=` — 預測結果（模組三產出）。

### 4.2 前端視覺化（依實用度排序）
1. **即時人數 + 現在「相對忙碌程度」**（vs 同時段歷史平均）。
2. **週熱力圖**：星期(7) × 小時(24) 的顏色矩陣 — 最直覺、最像 Google「熱門時段」。
3. **今日曲線**：實際值 + 預測值疊圖。
4. 日/週/月平均趨勢線。

### 4.3 聚合策略
- 即時查詢若每次掃全表會變慢 → 建立**預聚合表/物化視圖**（每小時 bucket 的平均），加速 dashboard。

## 5. 模組三：人數預測（ML）

### 5.1 先定義清楚「要預測什麼」
建議兩個產品分開做：
- **任務 A（短期）**：預測「**未來 1–3 小時**」的人數（給「現在要不要去」用）。
- **任務 B（整日）**：預測「**明天/指定日整日曲線**」（以小時或半小時為單位，給規劃用）。

### 5.2 特徵（features）
| 類別 | 特徵 |
|------|------|
| 時間 | 小時、半小時 bucket、星期幾、月份、是否週末 |
| 學期 | 是否上課日 / 寒暑假、學期第幾週、**考試週**、國定假日、補課日 |
| 近期 | 前 1 小時人數、昨天同時段、上週同星期同時段（lag 特徵） |
| 外部（加分） | 天氣（氣溫/降雨）、是否有校內大型活動 |

> 學校行事曆需要一份對照表（開學日、考試週、假期），可手動維護成一個小 CSV，價值很高。

### 5.3 模型路線（由簡到繁，逐步驗證）
1. **Baseline 0**：全體平均（最笨基準，當作及格線）。
2. **Baseline 1**：同星期同時段歷史平均（很強、常常打敗複雜模型）。
3. **進階**：`LightGBM` 回歸，吃上面所有特徵 — 通常 CP 值最高。
4. **時間序列法**：Prophet / SARIMA（可解釋、好處理週期）。
5. 資料夠多再考慮深度學習（LSTM/Temporal Fusion）— **非必要不上**。

### 5.4 評估
- 指標：**MAE**（好解讀，「平均差幾人」）、RMSE（懲罰大誤差）。
- **時間序列切分**：用「過去訓練、未來測試」的 rolling/expanding 切分，**絕不可隨機 shuffle**（會洩漏未來資訊）。
- 每個進階模型都要**贏過 Baseline 1** 才算有價值。

### 5.5 資料量門檻
- 週期模式至少要 **3–4 週**才看得出星期效應；學期/假期效應要**跨整個學期**才學得到。
- 先讓收集器穩定跑著養資料，預測模組可以晚一點再啟動。

## 7. 風險與注意事項

- **法務/禮貌**：確認校方是否允許抓取；控制頻率、標示 User-Agent。若有公開 API 優先用。
- **網站改版**：HTML 解析容易因改版壞掉 → 寫解析的單元測試 + 失敗告警，能快速察覺。
- **資料缺漏**：網站維護/當機會造成空洞 → null 處理、分析時妥善 imputation。
- **隱私**：只收集「總人數」這種聚合數字，不涉及個人資料，相對單純。
- **維運成本**：正式服務要 24/7，先用免費層（GitHub Actions + 免費小主機）壓成本，流量大再升級。
